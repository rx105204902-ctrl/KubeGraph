## Context

当前前端已经具备图谱工作区、调试面板和路径分析等基础能力，但仍存在三个明显缺口：

- Graph 组件在真实数据与复杂交互下仍可能出现无法稳定渲染、节点拖拽体验不顺畅或缩放/平移校验不足的问题。
- 页面文案仍然散落在组件中，缺少统一的国际化资源管理，无法支持中英文切换与默认语言策略。
- 右侧详情面板的 `metadata` 目前以 JSON 文本方式展示，阅读门槛高，且难以扩展排序、筛选和字段结构化展示能力。

这次变更需要在不推翻现有 `React + TypeScript + Cytoscape.js` 架构的前提下，把图谱交互可靠性、i18n 体系和详情区可读性一起补齐。

## Goals / Non-Goals

**Goals:**
- 提升 Graph 渲染和交互可靠性，确保拖拽、缩放、平移和大图性能可预测。
- 建立可维护的 i18n 结构，支持中英文文案资源抽离与全局切换。
- 优化按钮视觉规范和右侧资源详情展示，提升阅读效率和扩展性。

**Non-Goals:**
- 不替换现有图渲染引擎。
- 不在本次变更中引入完整的后端国际化或服务端多语言接口。
- 不把详情面板升级为完整的数据表管理系统；本次只预留结构化扩展能力。

## Decisions

### 决策：Graph 交互修复优先在现有 Cytoscape 生命周期内完成
将继续基于当前 Cytoscape 组件修复 Graph 问题，而不是迁移到新的图引擎。重点包括：

- 检查并修复数据同步、布局切换、拖拽状态和选中状态之间的冲突。
- 显式验证节点拖拽、画布平移、缩放边界和重布局后的视口保留行为。
- 在大图下关闭高成本动画或高频重排，以减少卡顿。

**考虑过的备选方案：**
- 更换图引擎：已否决，范围过大且会打断当前工作区改造。
- 只做样式层修补：已否决，根因在于 Graph 组件状态与交互链路。

### 决策：i18n 采用前端资源字典 + 全局语言上下文方案
前端将新增统一的语言资源文件和轻量语言上下文，负责：

- 组织中英文翻译资源。
- 暴露 `t()` 或等效访问接口供组件消费。
- 支持全局语言切换并在会话中保持。
- 以浏览器语言或本地缓存作为默认语言来源。

**考虑过的备选方案：**
- 继续手工在组件中 `if/else` 写双语：已否决，难以维护且容易漏改。
- 引入重量级 i18n 框架：暂不优先，先用轻量方案满足项目规模。

### 决策：详情区 `metadata` 使用结构化键值表呈现
右侧资源面板将把 `metadata` 从原始 JSON 优先转换为结构化表格视图：

- 每行展示字段名和值。
- 支持字段对齐和长内容换行。
- 对嵌套字段保持可扩展表示形式，必要时可降级为格式化文本。
- 在结构上预留后续排序、筛选和字段折叠能力。

**考虑过的备选方案：**
- 继续只展示 JSON：已否决，可读性差。
- 一开始就做完整表格组件：暂不采用，优先做清晰的结构化展示。

### 决策：按钮样式统一为主按钮 / 次按钮 / 文本按钮三级体系
视觉上将按钮统一为清晰的层级体系，明确 hover、active、disabled 状态，并确保在中英文切换后长度变化下仍保持布局稳定。

**考虑过的备选方案：**
- 继续沿用当前零散按钮变体：已否决，缺少统一交互语义。

## Risks / Trade-offs

- [i18n 资源覆盖不完整] → 通过集中资源文件和验收清单，逐步消除硬编码文案。
- [Graph 交互修复引入新的状态复杂度] → 将交互状态与渲染状态分层管理，并通过浏览器验收验证缩放/平移/拖拽链路。
- [结构化 metadata 遇到复杂嵌套字段可读性下降] → 先展示一层关键字段，并对复杂值保留格式化文本回退。

## Migration Plan

1. 抽离前端文案与语言资源，建立语言上下文和默认语言策略。
2. 修复 Graph 交互链路，重点验证拖拽、缩放、平移与大图性能表现。
3. 优化按钮样式与右侧资源面板，替换 `metadata` 展示方式。
4. 在中英文模式和常见分辨率下执行回归验证。

## Open Questions

- 是否需要在首版 i18n 中把 debug 面板和错误日志文案也全部国际化，还是先覆盖面向用户的主界面文案？
- `metadata` 表格是否需要首版就支持字段搜索，还是仅预留结构接口后续补充？
