## 背景

该项目旨在将 Kubernetes 资源可视化为一张图，让用户能够从单一视图中理解控制、流量、配置、存储与运行时之间的关系。当前讨论已经收敛出首版范围：按命名空间探索、使用类型化边而不是泛化依赖边、从选定资源进行一跳扩展，以及通过详情面板查看节点信息和关系证据。

主要约束在于图复杂度、关系准确性和实现速度。在核心价值尚未验证之前，构建全集群图或引入通用图数据库都会增加额外复杂性。因此，首个实现需要证明：一个小而明确的类型化关系模型，在真实集群中是可理解且有用的。

## 目标 / 非目标

**目标：**
- 定义带有明确边类型和方向规则的首版关系模型。
- 交付按命名空间进行的图探索流程，从选定资源开始并按需扩展一跳邻居。
- 在首个版本中支持固定集合的内置 Kubernetes 资源类型和关系类型。
- 暴露资源详情和关系解释，使用户能够理解节点为何相连。
- 建立一种实现简单、且在概念验证后易于扩展的技术方案。

**非目标：**
- 在首个版本中渲染全集群范围的图。
- 推断通用 CRD 之间的关系。
- 在首个版本中提供实时推送更新。
- 自动进行深层多跳扩展或路径分析。
- 将图数据持久化到外部数据库中。

## 设计决策

### 决策：使用类型化关系图，而不是通用依赖图
系统将使用固定的边类型来表示关系：`owns`、`selects`、`references`、`mounts`、`runs_on` 和 `routes_to`。每条边都将包含 `type`、`category`、`reason` 和 `evidence`，以便用户理解该关系为何存在。

这能让图更可信、更易调试。通用依赖边虽然更容易渲染，但会掩盖控制、路由、配置和运行时关系之间的重要差异。

**考虑过的备选方案：**
- 使用单一且无类型的依赖边：已否决，因为会降低可解释性。
- 为每类资源定义完全自定义的关系名称：已否决，因为会让筛选与 UI 行为变得不一致。

### 决策：在后端标准化图数据
后端将构建标准化的 `Node` 和 `Edge` 对象，并通过 API 暴露它们。前端负责渲染和交互，但不会自行推断关系。

这样可以把 Kubernetes 相关逻辑集中在一处，避免在 UI 中重复实现关系解析。

**考虑过的备选方案：**
- 让前端从原始清单中推断关系：已否决，因为这会让 UI 逻辑与 Kubernetes 模式细节耦合。
- 一开始就使用图数据库：已否决，因为首版只需要按命名空间、按需查询的能力。

### 决策：将首版范围限制为命名空间探索加一跳扩展
首个版本将允许用户选择命名空间，搜索或选择起始资源，并按需扩展一跳邻居；不会预先渲染整个命名空间或全集群图。

这样可以降低图爆炸问题，并提供更易理解的探索体验。

**考虑过的备选方案：**
- 立即渲染命名空间中的所有资源：已否决，因为资源密集的命名空间会变得难以阅读。
- 先构建全集群拓扑：已否决，因为在价值尚未验证前会引入性能和可用性风险。

### 决策：采用适合 Kubernetes 发现和图渲染的最小技术栈
后端将使用 `Go`、`client-go` 和 HTTP API 层；前端将使用 `React`、`TypeScript` 和 `Cytoscape.js`，并搭配标准组件库。

`Go` 和 `client-go` 最适合 Kubernetes 资源发现与关系提取。相比偏向流程图的库，`Cytoscape.js` 更适合图可视化和布局。

**考虑过的备选方案：**
- `React Flow`：首版中不采用，因为当前核心问题是图浏览，而不是工作流编辑。
- 先做实时流式传输：已否决，因为请求驱动的探索方式已足够支撑概念验证。

### 决策：标准化首版支持的资源与关系范围
首个版本将支持以下资源类型：`Ingress`、`Service`、`Deployment`、`ReplicaSet`、`Pod`、`ConfigMap`、`Secret`、`PersistentVolumeClaim`、`PersistentVolume`、`ServiceAccount`、`Node` 和 `HorizontalPodAutoscaler`。

首个版本将支持以下关系规则：
- `Deployment -> ReplicaSet`，类型为 `owns`
- `ReplicaSet -> Pod`，类型为 `owns`
- `Service -> Pod`，类型为 `selects`
- `Ingress -> Service`，类型为 `routes_to`
- `Pod -> ConfigMap` / `Secret` / `PersistentVolumeClaim`，类型为 `mounts`
- `Pod -> ConfigMap` / `Secret` / `ServiceAccount`，类型为 `references`
- `Pod -> Node`，类型为 `runs_on`
- `HorizontalPodAutoscaler -> Deployment`，类型为 `references`
- `PersistentVolumeClaim -> PersistentVolume`，类型为 `references`

这个范围能够在不过度扩张首版的前提下，提供对排障与架构理解都很有价值的一组关系切片。

## 风险 / 权衡

- 某些 Kubernetes 模式下，关系推断可能不完整：通过将首版规则明确限定在支持范围内来控制风险。
- 在多次扩展后，资源密集命名空间仍可能产生噪声较大的图：通过限制每次只扩展一跳并支持按边类型筛选来缓解。
- 后端负责标准化会增加初始服务端职责：接受这一点，以保持 UI 逻辑简单且一致。
- 没有实时推送时，图数据可能短暂落后于集群状态：在概念验证阶段接受该限制，并按需刷新数据。
- 命名空间范围会降低对全集群的即时可见性：接受这一点，先验证图探索的可用性，再决定是否扩大范围。

## 迁移计划

1. 为支持的资源类型实现后端资源发现与图关系提取。
2. 使用标准化图载荷暴露命名空间、搜索、节点详情和邻居查询 API。
3. 构建前端的命名空间选择器、图画布、筛选器和详情面板。
4. 在真实命名空间中验证支持的关系路径，重点包括 `Ingress -> Service -> Pod` 和 `Pod -> ConfigMap/Secret/PersistentVolumeClaim/Node`。
5. 仅在概念验证确认图模型和交互方式确实有价值后，再进一步扩展范围。

## 开放问题

- 首版节点详情是否应在元数据、状态和清单内容之外，也包含 Kubernetes 事件。
- `StatefulSet`、`DaemonSet`、`Job` 和 `CronJob` 是应在概念验证后立即加入，还是合并到第一轮编码中。
- 关系筛选器默认应展示所有边类型，还是只展示与当前选中资源最相关的类型。
