## ADDED Requirements（新增需求）

### Requirement: 系统必须提供可切换的图谱调试模式
系统必须提供显式的 debug 模式，用于展示节点 ID、边类型、元素数量、异常计数及其他诊断信息，以便开发和排障。

#### Scenario: 用户开启 debug 模式
- **WHEN（当）** 用户或开发者启用 debug 模式时
- **THEN（则）** 系统必须在不破坏主图可用性的前提下展示与当前图谱相关的诊断信息

#### Scenario: 用户关闭 debug 模式
- **WHEN（当）** 用户或开发者关闭 debug 模式时
- **THEN（则）** 系统必须恢复到面向普通用户的简洁视图，并隐藏调试信息

### Requirement: 系统必须记录图谱加载与渲染诊断日志
系统必须记录图谱请求、规范化、布局执行、错误恢复和重试等关键阶段的诊断日志，以支持异常排查。

#### Scenario: 图谱完成一次成功加载
- **WHEN（当）** 图谱从请求到渲染完成一次完整流程时
- **THEN（则）** 系统必须记录关键阶段的耗时或状态信息，供后续排查使用

#### Scenario: 图谱加载或布局失败
- **WHEN（当）** 图谱请求失败、规范化失败或布局失败时
- **THEN（则）** 系统必须记录失败阶段、原因摘要和是否触发重试或回退策略

### Requirement: 系统必须提供前端性能观测指标
系统必须提供图谱相关的性能指标，至少包括渲染耗时和 FPS 采样，并允许在 debug 模式下查看。

#### Scenario: 用户查看性能指标
- **WHEN（当）** debug 模式下图谱处于可交互状态时
- **THEN（则）** 系统必须展示当前或最近一次渲染相关的关键性能指标

#### Scenario: 图谱进入高负载状态
- **WHEN（当）** 当前图谱规模或交互频率导致性能下降时
- **THEN（则）** 系统必须能够显示性能下降迹象，帮助定位瓶颈来源

### Requirement: 系统必须为加载、空白和错误状态提供友好反馈
系统必须以用户可理解的方式展示加载中、无数据和异常失败状态，并为失败场景提供重试机制。

#### Scenario: 图谱正在加载
- **WHEN（当）** 图谱请求或布局尚未完成时
- **THEN（则）** 系统必须展示 Skeleton 或等效的加载占位，而不是空白画布

#### Scenario: 图谱返回空结果
- **WHEN（当）** 当前条件下没有可显示的节点或边时
- **THEN（则）** 系统必须展示清晰的空状态文案，并提示用户可以调整的方向

#### Scenario: 图谱失败后用户重试
- **WHEN（当）** 图谱加载失败且用户点击重试时
- **THEN（则）** 系统必须重新触发对应加载流程，并更新状态反馈而不是停留在原错误视图
