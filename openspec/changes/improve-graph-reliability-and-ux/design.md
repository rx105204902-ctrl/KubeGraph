## Context

当前图谱前端以 `App.tsx` 统一管理查询、选择、详情和错误状态，并通过 `GraphCanvas.tsx` 将数据一次性转换为 Cytoscape 元素。现有实现已能展示基础关系图，但仍存在几个结构性问题：

- 图谱数据进入渲染层前缺少稳定的规范化与异常隔离，容易因为重复元素、无效边或频繁全量更新造成节点/边丢失与布局错乱。
- `GraphCanvas` 在元素变化后会重新清空并执行布局，随着数据量上升会导致明显卡顿、闪烁和视口跳动。
- 页面布局仍偏向概念验证阶段，主操作路径分散在左侧查询、中央图布和右侧详情之间，缺少统一的工具栏、状态反馈和主题体系。
- 图谱缺少调试视图、性能观测和异常恢复路径，使“渲染失败”“元素缺失”“性能下降”等问题难以定位。

本次设计需要在保留现有 `React + TypeScript + Cytoscape.js` 技术栈的前提下，提升图谱的稳定性、交互能力和可维护性，并尽量避免对现有后端图谱契约做破坏性调整。

## Goals / Non-Goals

**Goals:**
- 建立稳定的图谱渲染输入管线，保证异常元素不会拖垮整张图。
- 将图谱页面重构为更清晰的工作区布局，并支持响应式和暗黑模式。
- 提供布局切换、搜索过滤、路径分析、子图展开/折叠、节点高亮和 tooltip 等探索能力。
- 为图谱增加 Skeleton、空状态、错误提示、重试、调试模式、日志和性能指标。
- 在主流浏览器和常见分辨率下保持一致的可用性与视觉层级。

**Non-Goals:**
- 不替换 `Cytoscape.js` 为其他图引擎。
- 不在本次设计中引入实时协同、多用户批注或完整的多跳图算法平台。
- 不重写后端全部图关系提取逻辑；仅在新增前端能力确有必要时扩展接口。
- 不在本次变更中引入重量级外部监控平台。

## Decisions

### 决策：引入图谱视图模型与规范化渲染管线
在图谱进入 `GraphCanvas` 之前，前端将新增一层稳定的视图模型处理：

- 对节点和边按稳定 ID 去重。
- 校验边的 `source` / `target` 是否存在，异常边隔离并记录诊断信息。
- 分离“原始图数据”“当前可见子图”“高亮/选择状态”“调试信息”等不同层级数据。
- 仅将经过规范化的数据发送给 Cytoscape。

这样可以把“数据问题”和“渲染问题”分开处理，避免因为单个异常边或重复元素导致整张图失败。

**考虑过的备选方案：**
- 继续让 `GraphCanvas` 直接消费 API 原始载荷：已否决，因为会把数据修复逻辑与渲染逻辑耦合在一起。
- 把所有规范化逻辑都放在后端：部分接受，但首版仍需要前端处理交互状态、筛选和子图视图。

### 决策：采用“工作区壳层 + 图引擎”分层结构
页面将拆分为更明确的结构：

- 顶部或侧边工具区：命名空间、搜索、布局切换、主题、调试、路径分析入口。
- 中央主视图：图谱画布、加载态、空状态与错误覆盖层。
- 右侧详情区：节点详情、边关系说明、路径结果和调试面板。

`App` 负责工作流状态与数据请求，图谱相关的派生状态通过独立 hooks 或状态模块管理，`GraphCanvas` 仅负责 Cytoscape 生命周期、批量更新与交互事件。

**考虑过的备选方案：**
- 继续在 `App.tsx` 中集中管理所有逻辑：已否决，因为已开始出现状态耦合和职责混杂。
- 直接引入完整状态管理框架：暂不采用，先用轻量 hooks 和上下文满足需求。

### 决策：布局与更新采用自适应策略，而非单一 force layout
图谱将支持 `force`、`hierarchical`、`circular` 等布局选项，并引入按数据规模选择策略：

- 小规模数据默认允许更细腻的布局与动画。
- 中大规模数据关闭高成本动画，优先使用更确定性的布局参数。
- 当布局执行失败、超时或元素规模超过阈值时，回退到更轻量的布局。
- 非显式切换布局时，筛选和高亮优先复用当前视口，避免频繁全量 `fit`。

**考虑过的备选方案：**
- 始终使用 `cose`：已否决，因为在大图下成本较高，且结果不稳定。
- 每次更新都重新执行全量布局：已否决，因为会造成视口跳动和交互中断。

### 决策：将“选择”与“展开”解耦，降低误操作和图爆炸
当前节点点击同时触发选择与扩展，容易导致用户只是想查看详情却意外扩张图规模。新设计将把这两个动作拆开：

- 点击节点默认进入选中与高亮。
- 通过明确的“展开邻居”“折叠子图”“路径分析起点/终点”动作执行结构变化。
- hover 展示轻量 tooltip，selected 状态展示更完整上下文。

**考虑过的备选方案：**
- 保留点击即展开：已否决，因为大图场景下容易失控。
- 使用双击展开：暂不采用，因为可发现性较弱，且触屏体验一般。

### 决策：内建轻量可观测能力，而不是依赖外部平台
本次变更将内建前端可观测能力：

- 图谱加载、规范化、布局执行、重试与失败原因的调试日志。
- Debug 模式下展示节点 ID、边类型、元素数量和异常计数。
- 通过 `requestAnimationFrame` 采样 FPS，通过时间戳统计数据处理和布局耗时。
- 面向用户展示 Skeleton、空状态、错误提示与重试动作；面向开发者展示可展开的调试信息。

**考虑过的备选方案：**
- 仅使用浏览器控制台临时调试：已否决，因为无法支撑持续定位问题。
- 接入完整 APM/监控平台：暂不采用，超出本次范围。

### 决策：建立统一设计令牌与主题体系
样式层将从当前散落的面板样式，升级为可复用的设计令牌（颜色、间距、字体、圆角、阴影、状态色），并支持暗黑模式和分辨率适配。布局以卡片式工作区为主，同时保证 1366 宽度笔记本和大屏场景的关键操作区始终可见。

**考虑过的备选方案：**
- 只在现有 CSS 上补零散样式：已否决，因为难以形成统一视觉系统。
- 直接引入完整 UI 框架：暂不采用，优先保持现有轻量前端栈。

## Risks / Trade-offs

- [状态层次增多] → 通过拆分 `graph data`、`workspace ui`、`debug state` 等 hooks，控制单个模块复杂度。
- [性能监控本身带来开销] → 默认低频采样，仅在 debug 模式展示详细指标。
- [显式展开/折叠会增加一个操作步骤] → 通过清晰按钮、右键/快捷动作和上下文面板降低感知成本。
- [更多布局选项可能让用户困惑] → 提供合理默认布局，并保留最近一次选择。
- [开放跨浏览器适配会拉长验证周期] → 优先验证 Chrome / Edge / Safari 常见分辨率，再扩展到其他环境。

## Migration Plan

1. 先补充规范文档与 capability specs，明确渲染稳定性、UI、探索和可观测需求。
2. 拆分前端状态结构，建立图谱视图模型和工作区壳层。
3. 重构 `GraphCanvas` 更新机制，引入布局切换、回退策略和增量更新。
4. 增加搜索过滤、路径分析、展开/折叠、tooltip 和高亮交互。
5. 增加 Skeleton、空状态、错误重试、debug 模式、日志和性能指标。
6. 在主流浏览器与常见分辨率下回归验证，并按结果微调默认布局和信息密度。

如需回滚，可先保留新的数据规范化层和状态管理，但隐藏高级功能入口，退回到单布局和基础交互模式。

## Open Questions

- 最短路径分析应完全在前端基于当前子图完成，还是在图规模变大后增加后端路径接口？
- Debug 模式是否需要对最终用户开放，还是仅在开发环境或显式开关下出现？
- 子图折叠是否需要持久化到会话状态，以避免刷新后丢失当前工作上下文？
